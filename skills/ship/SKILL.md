# ship — 一键发布流程（通用版）

> 本 skill 是**项目无关的流程骨架**。项目专属的命令/地址/测试矩阵全部来自项目内的
> `docs/SHIP-PROFILE.md`（部署档案）。**禁止把某个项目的命令套到另一个项目上。**

## 第 0 步：读取项目部署档案（强制前置）

1. `Read docs/SHIP-PROFILE.md`（项目根相对路径）
2. 档案不存在 → **停下**，按 [`PROFILE-TEMPLATE.md`](PROFILE-TEMPLATE.md) 引导创建（问王老师拿部署方式），创建后再继续
3. 档案里若声明"项目专属铁律文件"，一并 Read

档案提供：构建/类型检查命令、测试矩阵（全绿定义）、测试环境部署命令+地址、生产发版流程、GitHub 远程/分支、项目专属铁律指针。

## 通用铁律（所有项目恒真）

### ⛔ 铁律 A：开发前必做"业务视角分析"
任何代码改动前：① 一句话用户场景（xxx 角色做 xxx 动作期待 xxx 结果）；② 列 3~5 条用户视角不变量（上游/主档不能变、已生效状态不能倒退、审计轨迹必须保留）；③ 对照不变量反向审查方案。**用户视角优先于代码视角**——不看"字段 WHERE 怎么匹配"，看"删它对用户意味着什么"。

### ⛔ 铁律 B：拒绝点状修复，必做横向同类扫描
修 bug / 加能力前：想清楚这是哪一**类**行为 → grep 全仓库所有同类入口（同名方法/API/业务词）→ 每个入口按同一标准评估 → **一次性全改完** → 守卫测试覆盖整类（`test.each(入口清单)`）。点状修复 = 留坑给用户当测试员。

### ⛔ 铁律 C：协议类改造必做"影响路径盘点"
凡改数据交换协议（API body 结构、字段增删、保存格式、传参约定）：grep 所有调用方 → 每个调用点追溯数据源 → **影响路径清单打印给王老师确认**后才动代码。

### ⛔ 铁律 D：删除类操作必做"严谨引用审计"
删页面/API/组件/权限点/表：区分列表页 vs 子路径引用；查 router.push / href / Link / window.open / 菜单配置 / 外部 API 调用 / ORM 间接引用（级联、聚合）；**审计表打印给王老师确认**后才执行删除。

### ⛔ 铁律 E：ORM 字段先核对 schema
写 ORM 查询涉及字段名时，先 Read schema 定位 model 逐字确认。字段不存在**不要按习惯猜**（"A 表有所以 B 表也有"是最大坑，运行时才炸，TS 编译抓不到）。

### ⛔ 铁律 F：用户可见文案禁止裸英文枚举/裸 ID
列表状态、弹窗、Toast、API 错误 message 里不得出现 snake_case 枚举或 ID 片段，必须走项目的状态中文映射函数。修一处必横向扫一类（铁律 B）。

### ⛔ 铁律 G：失败自循环（Loop to Green）
测试/构建/类型检查失败时**不停下来等"继续"**：① 一次收集全部红灯 → ② 逐项诊断+最小修复（不为过测试改坏业务）→ ③ 重跑，每轮记一行进度。只在硬边界唤醒王老师：连续 2 轮红灯不降 / 需要业务决策 / 需要生产凭证 / **测试预期本身要改（必须王老师拍板）**。

## 主流程（严格按顺序，命令全部来自 SHIP-PROFILE）

1. （协议类）影响路径盘点 →王老师确认；（删除类）引用审计 → 王老师确认
2. 实施代码修改（守铁律 A/B/E/F）
3. **测试矩阵全绿**（档案定义的全部层级 + 类型检查 + lint；任一失败 → 铁律 G 循环，禁止带红发版）
4. 改了 API 行为 → 对应层测试**新增本轮断言**再跑；改了 UI → 按档案的 UI 验证方式补充
5. 同步文档（档案声明的：PRD / CHANGELOG / _INDEX 等）
6. **部署测试环境**（档案命令）→ **业务链路实地验证**：主场景 + 变体场景各走一遍，用 DB 查询 / curl / 日志取证。**代码合规 ≠ 业务闭环，绿灯 ≠ 完成**
7. 提交 GitHub：`git add` 相关文件（排除产物目录）→ conventional commit → push（档案分支）
8. **生产发版**（档案流程），强制前置：
   - 先查生产当前部署版本（让王老师跑 `git log -1` 或按档案方式确认 HEAD 与 pull 范围），不想当然"直接拉最新"
   - 涉及生产数据写操作 → **先备份**（mysqldump 等），基于真实 tool_result 核查备份存在，绝不虚构
9. 生产冒烟：档案声明的健康检查 + 关键业务动作一遍
10. 输出交付汇总报告（模板见下）

## 📋 交付汇总报告模板

```markdown
## Ship 报告
### 影响路径盘点 / 引用审计（协议/删除类必填，否则标"不适用"）
### 测试矩阵：各层结果（suites/tests 数）
### 部署：commit `xxx` → 测试环境地址 / 生产环境地址
### 业务链路实地验证
| 场景 | 操作 | 验证手段 | 结果 |
### ⚠️ 已知边界 / 风险（必写，禁止隐瞒未覆盖场景）
### 📝 GitHub：commit → 分支
```

## 硬约束

- 禁止用"代码合规"代替"业务验证"；没走业务链路验证禁止说"完成"
- 禁止隐瞒未覆盖场景——必须写进"已知边界"
- 删除/协议类操作禁止跳过王老师确认
- `$ARGUMENTS` 若提供，作为 commit message
